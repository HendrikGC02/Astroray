# Repository Cleanup & Professionalism Audit

> **Audience:** Maintainers and coding agents.  
> **Purpose:** Identify every gap between the current state of the repo and a
> clean, professional, agent-ready open-source project.  
> Each item is categorised by impact and includes a concrete action.

## 1. Root Directory Clutter

The project root currently contains a large number of files that do not belong there.

### 1a. Jupyter Notebooks / Exploration Scripts

| File | Size | Problem |
|---|---|---|
| `Extensive_testing_notebook.ipynb` | 11 MB | Dev exploration notebook; embedded cell outputs bloat git history |
| `Extensive_testing_notebook.py` | 29 KB | Auto-exported duplicate of the above |
| `Testing_python_package.ipynb` | 3 MB | Another dev notebook with embedded outputs |
| `Testing_python_package-S-5CG52530L0.ipynb` | 724 KB | Same notebook with a machine-specific device ID in the filename |

**Action:** Move to a `notebooks/` directory if they should be kept at all.
Strip cell outputs with `jupyter nbconvert --clear-output` before committing.
Add `.ipynb_checkpoints/` to `.gitignore`.  
The machine-ID notebook (`*-S-5CG52530L0.ipynb`) should simply be deleted — it is a local artefact.

### 1b. Generated Sample Output Images

| File | Size | Problem |
|---|---|---|
| `cornell_box.png` | 1.4 MB | Render output, not source; regenerated by tests |
| `material_test.png` | 940 KB | Same — regenerated by tests |
| `sphere_orbit.gif` | 4.3 MB | Large animation artefact |

**Action:** Delete from the repository and from git history (use `git filter-repo` or BFG).
Add `*.gif` and remove the output images from the root.
A `gallery/` or `docs/images/` folder linked from the README is acceptable for
showcase images that are intentionally small and curated.

### 1c. Misplaced PRD / Design Documents

|---|---|

| `PRD_render_bugs.md` | Same — a historical bug-fix PRD, belongs in `docs/` |

**Action:** `git mv PRD.md docs/PRD.md` and `git mv PRD_render_bugs.md docs/PRD_render_bugs.md`.
Update any cross-references.

### 1d. Dev Tooling Scripts

| File | Problem |
|---|---|
| `ralph_loop.sh` | 418-line autonomous aider loop referencing a local vLLM server. Highly environment-specific. |
| `build_cuda.bat` | Windows-only manual build helper; partially superseded by CMake |

**Action:** Move these to a `scripts/` or `tools/` directory and document their
prerequisites.  If `ralph_loop.sh` is only relevant to a specific developer

### 1e. Agent Configuration Files Mixed With Project Files

## 2. Large Binary / Generated Files Tracked in Git

Git is not well-suited for large binary files. Every clone, CI run, and agent
session pays the download cost.

| File | Size | Problem |
|---|---|---|
| `samples/test_env.hdr` | 6.2 MB | Runtime asset; not source code |
| `blender_addon/Test_scene.blend` | 764 KB | Binary Blender scene |
| `blender_addon/Test_scene.blend1` | 764 KB | Blender auto-backup — **must never be committed** |
| All notebooks | 14 MB total | Embedded output cells |

**Actions:**
- Add `*.blend1` to `.gitignore` immediately.
- Move `samples/test_env.hdr` to a release asset or use a download script; document this in the README.
- Consider [Git LFS](https://git-lfs.com/) for any binary assets that genuinely
  must stay in the repo (e.g. the `.blend` scene used by integration tests).
- Use `git filter-repo --strip-blobs-bigger-than 500K` (or BFG) to remove
  historical large blobs from git history.

## 3. Missing Professional Files

These files are expected in any mature open-source project and are also used by
coding agents to understand contribution rules, dependencies, and history.

| Missing File | Why It Matters |
|---|---|
| `LICENSE` | README claims MIT but no `LICENSE` file exists; legally unclear |
| `CONTRIBUTING.md` | Explains how to build, test, and submit PRs |
| `CHANGELOG.md` | Tracks breaking changes between versions |
| `.github/workflows/ci.yml` | Automated CI — builds and runs tests on every PR |
| `requirements.txt` or `pyproject.toml` | Lists Python test dependencies (`pytest`, `numpy`, `matplotlib`) so `pip install -r requirements.txt` sets up the test environment |
| `.editorconfig` | Enforces consistent indentation/line-endings across editors |

**Actions:**
1. Create `LICENSE` with the MIT text, year, and author name.
2. Create `requirements.txt` (at minimum `pytest numpy matplotlib`).
3. Create a minimal `CONTRIBUTING.md` pointing at the build steps in the README.
4. Add a GitHub Actions workflow that runs `cmake .. && make -j4 && pytest tests/ -v` on Ubuntu.

## 4. Documentation Structure

The `docs/` directory is a mix of internal agent plans, renderer reference
material, and implementation specs. This is useful, but needs clearer
organisation and navigation.

### Current state

```
docs/
  CUDA_IMPLEMENTATION_PLAN.md    (643 lines — agent task spec)
  GR_INTEGRATION_PLAN.md         (586 lines — agent task spec)
  HDRI_IMPLEMENTATION_PLAN.md    (650 lines — agent task spec / cline rules)
  agent-context/
    lessons-learned.md           (146 lines — rendering pitfalls log)
    renderer-internals.md        (292 lines — architecture reference)
```

### Problems

- No `docs/README.md` or `docs/index.md` — an agent starting fresh has no map.
- The three `*_PLAN.md` files are completed (their features ship in the current
  codebase) but are labelled as plans. They function as specs/ADRs now.
- `HDRI_IMPLEMENTATION_PLAN.md` doubles as a Cline rules file (`.clinerules`
  format inside a markdown doc) — confusing.
- `renderer-internals.md` is excellent but not linked from `README.md` or
  `AGENTS.md`.

### Actions

1. Add `docs/README.md` with a table of contents linking every doc.
2. Rename completed plan files to `docs/adr/` (Architecture Decision Records)
   or to `docs/CUDA_DESIGN.md`, etc., to reflect their archival nature.
3. Extract the embedded Cline rules from `HDRI_IMPLEMENTATION_PLAN.md` into
   `.clinerules.md` (or remove them; they duplicate `docs/agent-context/`).
4. Add a link from the main `README.md` to `docs/agent-context/renderer-internals.md`.

overlaps with what is in `docs/QUICKSTART.md` (referenced from AGENTS.md but

**Actions:**

  `AGENTS.md` — that is exactly what it is for.

  one-line reference in `AGENTS.md`.

  reference.

## 7. Code Organisation Issues

### 7a. `.c` File Inside `include/`

`include/stb_image_write_impl.c` is a C implementation file living in the
`include/` directory alongside headers. This is unconventional and can confuse
tools that scan headers.

**Action:** Move to `src/stb_image_write_impl.c` and update `CMakeLists.txt`.

### 7b. GPU Headers for Unfinished Feature

`include/astroray/` contains a set of CUDA GPU headers
(`gpu_bvh.h`, `gpu_materials.h`, `gpu_renderer.h`, `gpu_scene_upload.h`,
`gpu_types.h`) that correspond to in-progress feature work. They are stubs that
the current CPU build does not use.

**Action:** Document this clearly (a comment in each file or a `include/astroray/README.md`)
so agents don't assume these headers are functional or authoritative.

## 10. CI/CD

The project has no automated CI. Every PR lands without any build or test
validation. This is the highest-leverage professionalism gap.

**Minimum viable CI** (`.github/workflows/ci.yml`):

```yaml
on: [push, pull_request]
jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Install dependencies
        run: sudo apt-get install -y cmake libpython3-dev python3-pip &&
             pip install pytest numpy matplotlib
      - name: Build
        run: mkdir build && cd build && cmake .. -DCMAKE_BUILD_TYPE=Release && make -j4
      - name: Test
        run: pytest tests/ -v
```
